Создаю абстракцию файлового хранилища: порт `IFileStorageService` в domain (чистый TS) + локальный адаптер в infrastructure (Node fs), который складывает файлы в `apps/web/public/uploads/` — Vite раздаёт `public/` автоматически, так что файлы сразу доступны в браузере по URL. Переезд на S3 позже = один новый адаптер + одна строка в composition root.

⚠️ По сравнению с предыдущей версией плана — **переименование `IImageStorageService` → `IFileStorageService`** (более общий контракт, пригодится для любых файлов, не только картинок). Соответственно переименованы файл порта, файл адаптера, класс адаптера и синглтон в composition root.

## Контракт порта (минимум, Buffer)

```ts
export interface IFileStorageService {
  /** Сохраняет байты, возвращает публичный URL. ext — напр. "webp" или ".jpg". */
  save(buffer: Uint8Array, ext: string): Promise<ImageUrl>
  /** Удаляет файл по ранее возвращённому URL. */
  delete(url: ImageUrl): Promise<void>
}
```

Использую `Uint8Array` (встроенный JS-тип), не `Buffer` (Node-specific) — держит доменный контракт framework-agnostic и лучше мапится на AWS SDK в будущем. `Buffer` — подтип `Uint8Array`, прокидывается без конвертации.

Возвращаемый тип пока остаётся `ImageUrl` (= `string`, из `@panda-lavanda/shared`) — текущий потребитель — продукт (`images: ImageUrl[]`), а тип это всё равно просто алиас строки. Если позже появится общий `FileUrl`, заменим в одном месте. Это отдельное решение, сейчас не трогаю.

### 2. `packages/domain/src/index.ts` (правка)
Добавить `export * from './ports/file-storage.port'` рядом с двумя существующими портами.

### 3. `packages/infrastructure/src/storage/local-file-storage.service.ts` (новый)
Класс `LocalFileStorageService implements IFileStorageService`:
- Конструктор: `private readonly config: { storageDir: string; publicPath: string }` (например, `<web>/public/uploads` и `/uploads`).
- `save()`: `crypto.randomUUID()` + нормализованное расширение → `fs.mkdir(storageDir, { recursive: true })` → `fs.writeFile(...)` → вернуть `${publicPath}/${filename}`.
- `delete()`: извлечь basename из URL, проверить что он внутри `storageDir` (защита от path traversal), `fs.unlink` (игнорировать «файл не найден»).
- Импорты: `node:fs/promises`, `node:path`, `node:crypto`. Паттерн класса как в `DrizzleProductsRepository`.
- Удалить `.gitkeep` (папка больше не пустая).

### 4. `packages/infrastructure/src/index.ts` (правка)
Добавить `export * from './storage/local-file-storage.service'`.

### 5. `packages/infrastructure/tsconfig.json` (правка)
Добавить `"compilerOptions": { "types": ["node"] }` — зеркалим `packages/db/tsconfig.json`. Без этого `node:fs`/`process` не резолвятся (корневой tsconfig ставит `"types": ["vite/client"]`, который *замещает* список). `@types/node` уже поднят в корень.

### 6. `apps/web/src/app/providers/storage.ts` (новый — composition root)
Устанавливаю паттерн DI для хранилища (папка providers/ сейчас пустая):
```ts
import path from 'node:path'
import { LocalFileStorageService } from '@panda-lavanda/infrastructure'
import type { IFileStorageService } from '@panda-lavanda/domain'

// apps/web/public/uploads — Vite раздаёт public/ автоматически.
export const fileStorage: IFileStorageService = new LocalFileStorageService({
  storageDir: path.resolve(process.cwd(), 'public/uploads'),
  publicPath: '/uploads',
})
```
Именно здесь через конфиг/ENV потом подключится S3. Тип экспорта — порт, не класс (всё приложение зависит от `IFileStorageService`).

## Что НЕ делается в этой задаче
- Не трогаю DB-схему — картинки уже хранятся как `jsonb` массив URL в `products.images`, адаптер просто возвращает URL, который туда запишется через существующий `IProductsRepository.update()`.
- Не создаю upload endpoint / серверную функцию / UI загрузки — это отдельная задача; адаптер готов к использованию из server functions или use cases.
- Не подключаю реальный вызов из роута/страницы — composition root создаёт готовый синглтон `fileStorage`, который use cases подключат позже.
