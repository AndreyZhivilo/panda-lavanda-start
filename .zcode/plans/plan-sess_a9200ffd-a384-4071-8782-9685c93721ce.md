# План: «out-of-stock товары в конце списка»

## Цель
Товары, которых нет в наличии (все `exemplars.in_stock = false`), всегда должны отдаваться в конце списка каталога. Реализовать по канонам Clean Architecture: бизнес-правило живёт в `domain`, SQL-реализация в `infrastructure`.

## Решения (согласовано)
- **Контракт:** `SortOrder` enum + `sort?: SortOrder` в `IProductFilters` — задел под будущие сортировки (по цене, новизне и т.п.).
- **Tie-breaker:** добавить поле `created_at` в таблицу `products` (миграция БД). Вторичная сортировка — `created_at DESC`.
- **SQL-реализация:** **раздельные запросы** products (с EXISTS-сортировкой и пагинацией) и exemplars (по `product_id IN (...)`), сборка в JS. Это решает и новую задачу, и старую проблему дублирования строк от LEFT JOIN.

## Архитектура изменений (по слоям снизу вверх)

### 1. Domain — `SortOrder` + расширение фильтров

**`packages/domain/src/products/sort-order.ts`** *(новый)*:
```ts
/**
 * Sort options for product lists.
 *
 * Implemented as a const-object + union (not `const enum`) for
 * `isolatedModules` / `verbatimModuleSyntax` compatibility (see `Size`).
 */
export const SortOrder = {
  OUT_OF_STOCK_LAST: 'out-of-stock-last',
} as const
export type SortOrder = (typeof SortOrder)[keyof typeof SortOrder]
```
Экспортируется через `packages/domain/src/products/index.ts`.

**`packages/domain/src/products/product-filters.ts`** *(правка)* — добавить:
```ts
import type { SortOrder } from './sort-order'

export interface IProductFilters {
  categoryId?: UniqueId
  page?: number
  pageSize?: number
  ids?: UniqueId[]
  sort?: SortOrder[]   // ← NEW: multiple keys allowed; applied left-to-right
}
```
`sort` — массив, а не одиночное значение, потому что бизнес-правила часто комбинируются (например, «out-of-stock last, затем по цене»). Сейчас поддерживается только `OUT_OF_STOCK_LAST`, но массив даёт задел без смены контракта.

### 3. Infrastructure — рефакторинг `getMany`

**`packages/infrastructure/src/repositories/drizzle-products.repository.ts`** *(правка в `getMany`)* — переписать на раздельные запросы:

```ts
async getMany(filters: IProductFilters): Promise<Paginated<IProduct>> {
  const page = ...; const pageSize = ...; const offset = ...

  // (1) Загружаем страницу products — БЕЗ join с exemplars.
  //     exists-in-stock вычисляется коррелированным подзапросом;
  //     если sort содержит OUT_OF_STOCK_LAST — добавляем в ORDER BY.
  const productRows = await this.db
    .select({
      id: productsTable.id,
      name: productsTable.name,
      description: productsTable.description,
      category: productsTable.categoryId,
      images: productsTable.images,
      createdAt: productsTable.createdAt,
    })
    .from(productsTable)
    .where(and(...this.buildConditions(filters)))
    .orderBy(...this.buildOrderBy(filters.sort))
    .limit(pageSize)
    .offset(offset)

  if (productRows.length === 0) {
    // второй запрос не нужен; count вернём всё равно
    const total = await this.countTotal(filters)
    return { items: [], total }
  }

  // (2) Загружаем exemplars для найденных product ids одним IN-запросом.
  const productIds = productRows.map((p) => p.id)
  const exemplarRows = await this.db
    .select()
    .from(exemplarsTable)
    .where(inArray(exemplarsTable.productId, productIds))

  // (3) count для пагинации — тоже в этом же вызове, параллельно через Promise.all с (1)+(2)
  const total = await this.countTotal(filters)

  return { items: this.mergeProducts(productRows, exemplarRows), total }
}
```

**Новые приватные методы:**
- `buildOrderBy(sort?: SortOrder[])` — возвращает массив `SQL`-выражений. Если есть `OUT_OF_STOCK_LAST`, первый элемент — `(SELECT EXISTS ...) DESC` через `sql` template, второй (всегда) — `productsTable.createdAt desc()`. Без sort — дефолт `createdAt desc()`.
- `countTotal(filters)` — вынесенный существующий `SELECT count(*) FROM products WHERE ...`.
- `mergeProducts(productRows, exemplarRows)` — замена старого `groupProducts`: раскладывает exemplars по product.id, сохраняя порядок `productRows` (важно — именно в порядке сортировки).

**Старый `groupProducts` удаляется** — он работал на JOIN-rows, теперь этого формата нет.

**EXISTS-подзапрос** — через raw SQL Drizzle:
```ts
const inStockExpr = sql<boolean>`exists (
  select 1 from ${exemplarsTable}
  where ${exemplarsTable.productId} = ${productsTable.id}
    and ${exemplarsTable.inStock} = true
)`
```

### 4. Application — use case пропускает `sort`

**`packages/application/src/products/get-products.use-case.ts`** *(без изменений)* — он уже просто передаёт `filters` в репозиторий. Контракт `IProductFilters` расширился, use case автоматически поддерживает `sort`. Никакого кода не добавляется (это и есть преимущество подхода «полный путь»).

### 5. Web — route применяет сортировку

**`apps/web/src/app/server-functions/products/products.functions.ts`** *(правка)* — серверная функция передаёт `sort: [SortOrder.OUT_OF_STOCK_LAST]` в use case:
```ts
const result = await new GetProductsUseCase(productsRepository).execute({
  page: data.page,
  sort: [SortOrder.OUT_OF_STOCK_LAST],

```

Никаких изменений в UI/пагинации — для пользователя сортировка «невидима», просто товары всегда идут в правильном порядке. Параметр не expose-ится в URL (пока захардкожен как бизнес-правило; если позже захотим дать выбор — вынесем в `validateSearch`).

## Что НЕ делаем (намеренно)
- **Не exposeим `sort` в URL/query** — это бизнес-правило по умолчанию, а не пользовательский выбор. Пока `OUT_OF_STOCK_LAST` — единственное значение, оно применяется всегда.
- **Не показываем визуальный индикатор out-of-stock отдельно** — UI из `catalog-page.tsx` уже помечает «нет в наличии» бейджем; порядок просто изменится.
- **Не добавляем другие сортировки** (по цене, имени) — это зазор §12 #7 (🟡), отдельная задача. Контракт (`SortOrder[]`) к ним готов.

## Риски и митигации
- **Существующие данные получат одинаковый `created_at`** (now() при миграции). Это означает, что первичная сортировка (по `in_stock`) работает, но tie-breaker по `created_at` внутри группы «all-equal-timestamp» отдаётся на откуп Postgres (физический порядок). При следующем seed скрипт insert-ит строки по одной — у каждой будет свой timestamp, и новизна будет корректной. Для прототипа допустимо.
- **N+1 exemplars** — mitigated: один IN-запрос по `product_id IN (page_ids)`, не по одному на товар.
- **Изменение `groupProducts` → `mergeProducts`** — поведение для `getById` тоже меняется (он использовал `groupProducts`). Нужно обновить `getById` аналогично: один SELECT products + один SELECT exemplars по id. В плане это учтено — `getById` будет переписан в том же стиле.
