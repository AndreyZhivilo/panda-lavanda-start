# Запуск в Docker

В этом проекте в Docker контейнеризована **только база данных PostgreSQL**.
Веб-приложение (`apps/web`) и telegram-бот (`apps/telegram-bot`) запускаются
на хосте через `npm run dev` и подключаются к БД на `localhost:5432`.

Дополнительно есть одноразовый сервис `migrate`, который применяет
Drizzle-миграции при старте.

## Быстрый старт

Четыре шага из корня репозитория:

```bash
# 1. (опционально) задать свои значения для БД. Без этого шага
#    сработают значения по умолчанию: panda / panda / panda
cp .env.example .env

# 2. Установить зависимости и сгенерировать SQL-миграции из схемы Drizzle.
#    Миграций в packages/db/drizzle по умолчанию нет — их нужно сгенерировать.
npm install
npm run generate -w @panda-lavanda/db

# 3. Поднять БД и применить миграции.
docker compose up -d

# 4. Запустить веб-приложение на хосте.
npm run dev
```

После четвёртого шага приложение доступно на **http://localhost:3000**.

│  └────────┬────────┘                └──────────┬───────────┘ │
│           │ DATABASE_URL                       │ DATABASE_URL │
│           │ = …@localhost:5432                 │              │
│           ▼                                    ▼              │
│  ┌─────────────────────────┐  :5432  ┌────────────────────┐  │
│  │  Docker                 │◄────────│  порт проброшен    │  │
│  │  ┌───────────────────┐  │         └────────────────────┘  │
│  │  │  postgres:17      │  │                                   │
│  │  │  (БД)             │  │                                   │
│  │  └────────▲──────────┘  │                                   │
│  │           │ db (имя сервиса во внутренней сети)            │
│  │  ┌────────┴──────────┐  │                                   │
│  │  │  node:22-alpine   │  │                                   │
│  │  │  migrate          │  │                                   │
│  │  │  (one-shot)       │  │                                   │
│  │  └───────────────────┘  │                                   │
│  └─────────────────────────┘                                   │
└──────────────────────────────────────────────────────────────┘

- **БД** (`postgres:17-alpine`) и **мигратор** (`node:22-alpine`) живут в Docker.
- **Веб** и **бот** запускаются локально и ходят в БД по `localhost:5432`.
- **Мигратор** внутри docker-сети обращается к БД по имени сервиса `db`.

## Переменные окружения

Переменные берутся из файла `.env` в корне репозитория. Если файла нет,
`docker-compose.yml` использует значения по умолчанию (`panda`/`panda`/`panda`).

| Переменная          | Назначение                          | По умолчанию |
| ------------------- | ----------------------------------- | ------------ |
| `POSTGRES_USER`     | Пользователь PostgreSQL             | `panda`      |
| `POSTGRES_PASSWORD` | Пароль PostgreSQL                   | `panda`      |
| `POSTGRES_DB`       | Имя БД, создаваемой при первом старте | `panda`    |
| `DATABASE_URL`      | Полная строка подключения           | `postgresql://panda:panda@localhost:5432/panda` |

### Важно: два значения `DATABASE_URL`

| Кто использует          | Хост в `DATABASE_URL` | Почему                                  |
| ----------------------- | --------------------- | --------------------------------------- |
| Команды на хосте (`npm run dev`, `drizzle-kit`) | `localhost` | Веб/бот вне Docker, стучатся на проброшенный порт |
| Сервис `migrate` в Docker | `db` | Внутри docker-сети БД доступна по имени сервиса |

В файле `.env` указывайте значение **для хоста** (`localhost`).
В сервисе `migrate` значение переопределяется внутри `docker-compose.yml`
(хост подменяется на `db`), править ничего не нужно.

## Работа с миграциями

Миграции — это файлы SQL в каталоге `packages/db/drizzle/`.
Их **две separate операции**:

### 1. Генерация (на хосте)

Сравнивает схему Drizzle (`packages/db/src/schema/`) с последней миграцией
и создаёт новый `.sql`-файл. **Не трогает БД.**

```bash
npm run generate -w @panda-lavanda/db
```

> В репозитории нет ни одной миграции, поэтому перед первым `docker compose up`
> обязательно выполните эту команду — иначе сервису `migrate` будет нечего
> применять.

### 2. Применение (в Docker, one-shot сервисом)

Запускается автоматически вместе с `docker compose up`. Применяет все
несозданные миграции к БД.

Чтобы запустить миграции вручную (без пересоздания БД):

```bash
docker compose run --rm migrate
```

### Типичный цикл изменения схемы

1. Меняете код схемы в `packages/db/src/schema/`.
2. `npm run generate -w @panda-lavanda/db` — появился новый `.sql`.
3. `docker compose run --rm migrate` — применили к работающей БД.

## Подключение к базе данных

### Через psql

```bash
docker exec -it panda-db psql -U panda -d panda
```

(замените `panda`/`panda` на свои `POSTGRES_USER`/`POSTGRES_DB`, если меняли).

Внутри `psql` полезные команды: `\dt` (список таблиц), `\q` (выход).

### Через Drizzle Studio

```bash
# на хосте, со строкой подключения из .env
npm run studio -w @panda-lavanda/db
```

Studio откроется в браузере на http://localhost:4979 (порт может отличаться).

### Через любой SQL-клиент (DBeaver, TablePlus, DataGrip)

| Параметр | Значение           |
| -------- | ------------------ |
| Host     | `localhost`        |
| Port     | `5432`             |
| User     | `panda`            |
| Password | `panda`            |
| Database | `panda`            |

## Полезные команды

Все команды — из корня репозитория.

```bash
# Поднять БД + применить миграции (в фоне)
docker compose up -d

# Посмотреть статус контейнеров
docker compose ps

# Логи БД
docker compose logs -f db

# Логи мигратора
docker compose logs migrate

# Остановить контейнеры (данные в volume сохранятся)
docker compose stop

# Остановить и удалить контейнеры (данные сохранятся в volume pgdata)
docker compose down

# Остановить, удалить контейнеры И данные БД (полный сброс)
# ⚠️ Это удалит все таблицы и данные!
docker compose down -v

# Применить миграции вручную (БД должна быть запущена)
docker compose run --rm migrate

# Пересоздать контейнер БД с нуля после down -v
docker compose up -d
```

## Решение проблем

### «Bind for 0.0.0.0:5432 failed: port is already allocated»

Порт 5432 занят — скорее всего, локальный PostgreSQL уже запущен.

**Варианты:**

1. Остановить локальный PostgreSQL (служба/`brew services stop postgresql`/Диспетчер служб).
2. Или поменять проброс порта в `docker-compose.yml`:
   ```yaml
   db:
     ports:
       - "5433:5432"   # наружу на 5433, внутри сети по-прежнему 5432
   ```
   Тогда `DATABASE_URL` для хоста станет `postgresql://panda:panda@localhost:5433/panda`.

### Сервис `migrate` падает с «No migrations found»

В `packages/db/drizzle/` ещё нет миграций. Сгенерируйте их:

```bash
npm run generate -w @panda-lavanda/db
docker compose run --rm migrate
```

### Ошибка аутентификации (`password authentication failed`)

Чаще всего — расхождение пароля в `.env` и в уже существующем volume.
Пароль `POSTGRES_PASSWORD` применяется **только при первом создании БД**.
Если вы сменили пароль после первого старта, нужно пересоздать том:

```bash
docker compose down -v   # ⚠️ удалит все данные БД
# привести .env в соответствие
docker compose up -d
```

### Хочу начать с чистого листа (полный сброс)

```bash
docker compose down -v   # удалить контейнеры + том с данными
docker compose up -d     # поднять заново
```

### Контейнер `migrate` не останавливается / перезапускается

По задумке он стартует, применяет миграции и завершается с кодом 0.
Если он постоянно перезапускается — посмотрите логи:

```bash
docker compose logs migrate
```

Чаще всего причина: БД ещё не готова (хотя `depends_on: service_healthy`
должен это покрывать) или нет миграций.

## Как позже контейнеризовать веб

Сейчас это не реализовано. Краткий план на будущее:

1. Добавить `apps/web/Dockerfile` с многоэтапной сборкой:
   - **builder**: скопировать репозиторий, `npm ci`, `npm run generate-routes`, `npm run build` (Vite/TanStack Start соберёт SSR-бандл в `.output/`).
   - **runner**: на базе `node:22-alpine`, скопировать только `.output/` + `node_modules` продакшн-зависимостей, запустить `node .output/server/index.mjs`.
2. В `docker-compose.yml` добавить сервис `web`, который зависит от `db` (healthy) и `migrate` (completed), получает `DATABASE_URL` с хостом `db`.
3. Веб на хосте при этом можно оставить для разработки, а контейнер — для продакшн-проверок.

Полностью актуальная архитектура слоёв и зависимостей описана в `architecture.md` и `AGENTS.md`.
